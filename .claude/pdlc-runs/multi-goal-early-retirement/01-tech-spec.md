## 1. Architecture overview

The Multi-Goal feature extends the existing retirement-planner as a **new tab** (`tab-multigoal.html`) with a dedicated calculation module (`calc-multigoal.js`).

**Persistence**: Phases stored in separate localStorage key `rp_phases`. Sharelink encoding OPTIONAL via checkbox (see Section 3 for full persistence strategy).

**Core flow**:
1. User configures life phases (age ranges + expenses + inflation) in the Multi-Goal tab UI
2. User clicks "Calculate Multi-Goal Projections"
3. Calculation engine:
   - Reads corpus accumulation result from existing Financial Plan / Projections tabs (via `RP._preReturn`, `RP._postReturn`, retirement-age corpus)
   - Allocates corpus across phases using PV-proportional algorithm (default) OR user-override percentages
   - Runs year-by-year projection: each phase's sub-corpus grows at post-return rate, depletes by phase's inflation-adjusted expenses
   - Detects deficits (phase runs out before phase end) and calculates actionable suggestions
4. Results rendered as allocation breakdown table + year-by-year projection table + chart with phase markers

**Isolation strategy**: Multi-Goal tab does NOT modify existing tabs. Existing Projections tab continues to use single-bucket `postRetireMonthly` model. User can compare both approaches side-by-side within the same session.

## 3. Data model

### Phase object schema
```javascript
{
  id: string,                    // unique ID (e.g., 'phase-1714156800000' from Date.now())
  name: string,                  // user-provided label (e.g., "Kids in college")
  startAge: number,              // absolute user age (e.g., 50)
  endAge: number,                // absolute user age (e.g., 54)
  baseMonthlyExpense: number,    // in ₹, today's value (e.g., 200000)
  inflationRate: number          // % per year (e.g., 10.0 for education inflation)
}
```

**Validation rules**:
- `startAge` >= `RP.val('retirementAge')` (phases start at/after retirement)
- `endAge` <= `RP.val('lifeExpectancy')`
- `endAge` > `startAge` (minimum 1-year duration)
- `inflationRate`: 0-20% range (matches existing `inflationRate` input validation)
- `baseMonthlyExpense` > 0
- **Overlaps ALLOWED**: Multiple phases can cover the same age year → sum expenses in projection loop
- **Gaps ALLOWED**: Years not covered by any phase → zero expense for that year + informational warning in UI

### Runtime state
```javascript
RP._phases = []                      // Array of phase objects (see schema above)
RP._multiGoalProjectionRows = []     // Projection results (see below)
RP._phaseAllocations = {}            // Computed allocation breakdown (for UI display)
```

### Projection result schema
```javascript
{
  age: number,                       // user's age for this row
  starting: number,                  // total corpus at start of year (sum across all phase buckets)
  growth: number,                    // total growth (sum of all phase buckets' growth)
  expenses: number,                  // total expenses withdrawn (sum across active phases)
  ending: number,                    // total corpus at end of year
  status: string,                    // 'Earning' | 'Retired' | 'Dead'
  activePhases: [                    // phases covering this age
    {
      phaseId: string,               // matches phase.id
      phaseName: string,
      bucketStarting: number,        // this phase's sub-corpus at start of year
      bucketGrowth: number,
      bucketExpense: number,         // inflated expense for this phase this year
      bucketEnding: number,
      isExhausted: boolean           // true if bucket hit zero mid-phase
    }
  ]
}
```

### Allocation breakdown schema (pre-flight display)
```javascript
{
  phases: [
    {
      phaseId: string,
      phaseName: string,
      ageRange: string,              // e.g., "50-54"
      pvRequired: number,            // Present Value of all future expenses for this phase
      allocated: number,             // Actual corpus allocated (proportional to PV)
      percentOfCorpus: number,       // % of total corpus (e.g., 42.5)
      deficit: number                // If pvRequired > allocated, else 0
    }
  ],
  totalPV: number,                   // Sum of all pvRequired
  totalCorpus: number,               // Available corpus at retirement
  surplus: number                    // If totalCorpus > totalPV, else 0
  overallDeficit: number             // If totalPV > totalCorpus, else 0
}
```

### Persistence strategy

**DECISION (Gate A 2026-04-27 — Option C)**: Separate localStorage key `rp_phases` AS PRIMARY storage; sharelink encoding is OPTIONAL via a "Include phases in shared link" checkbox in the share-link UI.

**Local persistence** (ALWAYS):
```javascript
// In calc-multigoal.js
RP.savePhasesData = function () {
  localStorage.setItem('rp_phases', JSON.stringify(RP._phases));
};
RP.loadPhasesData = function () {
  const saved = localStorage.getItem('rp_phases');
  return saved ? JSON.parse(saved) : [];
};
```

**Rationale**:
- **Matches existing convention** (verified via convention-checker agent):
  - Expense Tracker: separate key `rp_expense_log`
  - Net Worth Tracker: separate key `rp_networth_log`
  - Investment Tracker: separate keys `rp_tracker_entries`, `rp_tracker_mode`
  - Dark mode: separate key `rp_dark_mode`
- **Profiles are NOT modified** — zero changes to `profiles.js`
- **Profiles/sharelinks are INPUT-FIELDS-ONLY**: `getAllInputIds()` explicitly collects only `<input>` elements; this precedent is preserved

**Sharelink encoding** (OPTIONAL, user-controlled):
- When user clicks share-link button, check new "Include phases in shared link" checkbox (default: ON)
- If checked: append `&phases=<base64(JSON.stringify(RP._phases))>` to share URL
- If unchecked: share URL identical to today's behavior (input fields only)

**Sharelink decoding** (small modification to `sharelink.js`):
```javascript
// In RP.loadFromShareLink(), after existing input load:
const phasesData = params.get('phases');
if (phasesData) {
  try {
    RP._phases = JSON.parse(atob(phasesData));
    // Validate each phase (see data contracts doc for validation rules)
    RP._phases = RP._phases.filter(phase => RP.validatePhase(phase));
    RP.savePhasesData(); // Persist locally after successful import
    if (RP.renderPhases) RP.renderPhases(); // Render multi-goal tab UI
  } catch (e) {
    console.warn('Invalid phases data in share link, skipping:', e);
  }
}
```

**Backward compatibility**:
- Existing sharelinks without `phases` param → multi-goal tab starts empty (user in single-bucket mode)
- New sharelinks with `phases` → auto-populates multi-goal tab + persists locally
- Graceful degradation: malformed `phases` param logged to console + skipped (doesn't crash page)

**Wireframe update needed**: Add "Include phases in shared link" checkbox next to existing "Copy Share Link" button (see Design Lead for wireframe update in Stage 3)

## 7. Risks

### Risk 1: Persistence pattern ~~breaks existing convention~~ **RESOLVED via Option C**
- **Description**: ~~Intake assumption A3 proposes extending profiles/sharelinks~~ **RESOLVED (Gate A 2026-04-27)**: Option C chosen — separate localStorage key `rp_phases` AS PRIMARY (matches existing convention), sharelink encoding OPTIONAL via checkbox.
- **Likelihood**: N/A (decision locked)
- **Impact**: None — existing convention preserved for local persistence; sharelink extension is user-controlled opt-in
- **Mitigation**: 
  - **DONE**: Section 3 updated with Option C implementation
  - Validation contract added (see data contracts doc): imported phases MUST pass validation or be rejected with console.warn
  - Wireframe update flagged for Design Lead: add "Include phases in shared link" checkbox

### Risk 2: PV allocation math error compounds across 30+ years
- **Description**: PV-proportional allocation uses discounted cash flow calculation. Off-by-one errors in year indexing, wrong compounding direction (inflation vs discount), or FP rounding can cause silent 10-20% allocation errors.
- **Likelihood**: Medium (complex financial math, easy to get formula direction wrong)
- **Impact**: High — wrong allocation means user trusts incorrect retirement plan, potentially underfunding critical phases (e.g., medical)
- **Mitigation**:
  - **test-multigoal.html is MANDATORY** (per Q5 commitment) — 10-15 scenarios with hand-verified expected outputs BEFORE bug bash
  - Test cases MUST include:
    - Single phase (allocation = 100% of corpus)
    - Two phases, equal duration/expense, same inflation (allocation = 50/50)
    - Two phases, 10% inflation vs 6% inflation (verify higher-inflation phase gets MORE allocation due to PV)
    - Deficit scenario (total PV > corpus) — verify proportional scaling
    - Surplus scenario (total PV < corpus) — verify leftover corpus calculation
  - **Double-review of PV formula** during implementation by Tech Lead (me)

### Risk 3: Year-by-year projection loop doesn't match single-bucket behavior for edge cases
- **Description**: Existing calc-projections.js has edge-case handling (e.g., corpus goes negative, exact-age retirement). Multi-goal loop must handle same edges PLUS new ones (phase ends mid-year, bucket exhausted but other buckets remain).
- **Likelihood**: Medium
- **Impact**: Medium — wrong projection results, or JS errors on edge inputs
- **Mitigation**:
  - Copy year-loop structure from calc-projections.js:27-61 as starting point (don't reinvent)
  - Test scenarios MUST include:
    - Retirement age = current age (zero earning years)
    - Life expectancy = retirement age + 1 (one year retired)
    - Phase ending exactly at life expectancy
    - All phases exhausted before life expectancy (user outlives money)

### Risk 4: Chart rendering breaks on overlapping phases or >5 phases
- **Description**: Phase shaded regions drawn via Canvas `fillRect()` with 10% opacity. Overlapping phases stack opacity → visual confusion. 10+ phases → chart becomes unreadable rainbow.
- **Likelihood**: Low (most users will have 3-5 phases per intake example)
- **Impact**: Low — chart is supplementary; table data remains correct
- **Mitigation**:
  - Cap opacity at 10% regardless of overlap count (don't compound opacity)
  - If >7 phases, show warning: "Chart may be cluttered; refer to table for details"
  - Defer advanced chart (stacked area, zoomable timeline) to v2 unless requested

### Risk 5: Floating-point precision errors in long projections (30-50 years)
- **Description**: JavaScript uses IEEE 754 doubles. Repeated multiplication/division over 50 iterations can accumulate rounding errors (~₹100-1000 variance in crore-scale corpus).
- **Likelihood**: Medium (long time horizons are the feature's primary use case)
- **Impact**: Low — ₹1000 error in ₹5 Cr corpus is 0.02%, negligible for planning purposes
- **Mitigation**:
  - Use integer cents internally if errors observed in testing (multiply all ₹ by 100, divide on display)
  - Test 50-year projection (age 30 → 80) and verify total corpus delta vs hand calculation is <0.1%
  - Document acceptable precision: "Projections accurate to ±₹10,000 over 50 years"

### Risk 6: User configures nonsensical phases (overlapping, duplicate, out-of-order)
- **Description**: UI allows freeform input. User could add Phase A (50-60) then Phase B (55-65) → overlap. Or Phase C (70-75) then Phase D (60-65) → out-of-order.
- **Likelihood**: High (no validation during add, only on calculate)
- **Impact**: Low — confusing UX, but math handles overlaps correctly (sums expenses)
- **Mitigation**:
  - Auto-sort phases by `startAge` on render (matches intake assumption A14)
  - Show overlap warning badge: "Overlaps with Phase X" (informational, not blocking)
  - Show gap warning: "Ages 61-69: No expenses configured"
  - No validation errors — allow user flexibility (they may intentionally model overlapping costs)

### Risk 7: Existing tabs break if RP._postReturn is undefined
- **Description**: Multi-goal reads `RP._postReturn`, which is set by calc-financial-plan.js:48. If user never visits Financial Plan tab, `RP._postReturn` may be undefined → NaN in allocation math.
- **Likelihood**: Low (calculateAll() in app.js calls all calc functions on page load)
- **Impact**: High if it happens — NaN propagates through entire calculation
- **Mitigation**:
  - Verify app.js boot sequence: does calculateAll() run automatically? (Need to check app.js init)
  - Add defensive default in calc-multigoal.js: `const postReturn = RP._postReturn || 0.05;`
  - Log warning to console if fallback used: "Warning: Post-return rate not calculated, using default 5%"

## Appendix A: PV-Proportional Allocation Algorithm (Pseudocode)

*(Core math engine for Phase allocation — included in Tech Spec for reviewer verification)*

```javascript
/**
 * Allocate retirement corpus across life phases proportional to Present Value of each phase's future expenses
 * 
 * Inputs:
 *   - totalCorpus: ₹ available at retirement (from projections engine)
 *   - phases: Array of {startAge, endAge, baseMonthlyExpense, inflationRate}
 *   - retirementAge, currentAge: user ages
 *   - postReturn: blended post-retirement return rate (decimal, e.g., 0.05 for 5%)
 * 
 * Outputs:
 *   - allocations: { phaseId: ₹ allocated, ... }
 *   - pvRequired: { phaseId: ₹ PV, ... }
 *   - totalPV, surplus, deficit
 */
function calculatePVAllocation(totalCorpus, phases, retirementAge, currentAge, postReturn) {
  const pvRequired = {};
  let totalPV = 0;

  // Step 1: Calculate PV for each phase
  for (const phase of phases) {
    let phasePV = 0;
    const phaseInflation = phase.inflationRate / 100;

    for (let age = phase.startAge; age <= phase.endAge; age++) {
      // Years from NOW (for inflation calc — expense grows from today)
      const yearsFromNow = age - currentAge;
      
      // Inflated annual expense at this age
      const inflatedAnnual = phase.baseMonthlyExpense * Math.pow(1 + phaseInflation, yearsFromNow) * 12;
      
      // Years from RETIREMENT (for discounting — PV calc)
      const yearsFromRetirement = age - retirementAge;
      
      // Discount back to retirement age using post-return rate
      const discountFactor = Math.pow(1 + postReturn, yearsFromRetirement);
      const pvOfThisYear = inflatedAnnual / discountFactor;
      
      phasePV += pvOfThisYear;
    }

    pvRequired[phase.id] = phasePV;
    totalPV += phasePV;
  }

  // Step 2: Allocate proportionally
  const allocations = {};
  let totalAllocated = 0;

  if (totalPV > 0) {
    for (const phase of phases) {
      const proportion = pvRequired[phase.id] / totalPV;
      allocations[phase.id] = totalCorpus * proportion;
      totalAllocated += allocations[phase.id];
    }
  }

  // Step 3: Calculate surplus or deficit
  const surplus = Math.max(0, totalCorpus - totalPV);
  const deficit = Math.max(0, totalPV - totalCorpus);

  return { allocations, pvRequired, totalPV, totalAllocated, surplus, deficit };
}
```

**Key insights**:
1. **Inflation compounds from `currentAge`**, not `retirementAge` — expense grows over user's entire life
2. **Discount compounds from `retirementAge`** — PV is "value at retirement"
3. **If totalPV > corpus** (deficit), allocations scale down proportionally — each phase gets `(corpus / totalPV) * pvRequired[phase]`
4. **If totalPV < corpus** (surplus), leftover corpus remains unallocated (emergency buffer / legacy)

**Test scenarios for validation** (to be implemented in test-multigoal.html):
- Single phase, zero inflation → PV = baseMonthly × 12 × duration
- Two phases, equal PV → allocation = 50/50
- High-inflation phase vs low-inflation phase → high-inflation gets MORE allocation (PV math accounts for faster growth)
- Deficit scenario → verify each phase's allocation = proportion of available corpus

## Appendix B: Year-by-Year Projection Loop (Pseudocode)

```javascript
/**
 * Generate year-by-year projection with multi-phase expense depletion
 * Similar to calc-projections.js but with per-phase sub-corpus tracking
 */
function generateMultiGoalProjections(phases, allocations, retirementAge, lifeExpectancy, currentAge, postReturn, inflationRate) {
  const rows = [];
  
  // Initialize sub-corpus buckets at retirement
  const phaseBuckets = {};
  for (const phase of phases) {
    phaseBuckets[phase.id] = {
      balance: allocations[phase.id],
      exhaustedAt: null
    };
  }

  for (let age = retirementAge; age <= lifeExpectancy; age++) {
    const activePhases = phases.filter(p => age >= p.startAge && age <= p.endAge);
    
    let totalStarting = 0;
    let totalGrowth = 0;
    let totalExpenses = 0;
    const phaseDetails = [];

    for (const phase of activePhases) {
      const bucket = phaseBuckets[phase.id];
      if (bucket.exhaustedAt !== null) continue; // Skip exhausted buckets

      const starting = bucket.balance;
      const growth = starting * postReturn;
      
      const yearsFromNow = age - currentAge;
      const phaseInflation = phase.inflationRate / 100;
      const inflatedAnnual = phase.baseMonthlyExpense * Math.pow(1 + phaseInflation, yearsFromNow) * 12;
      
      const ending = starting + growth - inflatedAnnual;

      if (ending < 0 && bucket.exhaustedAt === null) {
        bucket.exhaustedAt = age;
      }

      bucket.balance = Math.max(0, ending); // Don't go negative

      phaseDetails.push({
        phaseId: phase.id,
        phaseName: phase.name,
        bucketStarting: starting,
        bucketGrowth: growth,
        bucketExpense: inflatedAnnual,
        bucketEnding: bucket.balance,
        isExhausted: bucket.exhaustedAt !== null
      });

      totalStarting += starting;
      totalGrowth += growth;
      totalExpenses += inflatedAnnual;
    }

    const totalEnding = Object.values(phaseBuckets).reduce((sum, b) => sum + b.balance, 0);

    rows.push({
      age,
      starting: totalStarting,
      growth: totalGrowth,
      expenses: totalExpenses,
      ending: totalEnding,
      status: age < lifeExpectancy ? 'Retired' : 'Dead',
      activePhases: phaseDetails
    });
  }

  return rows;
}
```

**Key differences from single-bucket calc-projections.js**:
- Multiple sub-corpus buckets (one per phase), tracked independently
- Each bucket grows + depletes separately → one phase can exhaust while others continue
- Total corpus = sum of all bucket balances (allows for "partial survival" — some phases funded, others not)
- `activePhases` filter for current age → handles overlaps (sum expenses) and gaps (zero expense) naturally
